## What Are Forms?

Forms provide a user-friendly interface for executing workflows. They:
- Collect user input with validation
- Display dynamic dropdown options
- Show/hide fields conditionally
- Execute workflows on submission
- Display results

## How They Work

```
User fills form → Form validates input → Workflow executes → Results displayed
```

**Form lifecycle**:
1. User navigates to form
2. Form loads with fields
3. Data providers populate dropdowns
4. User fills fields
5. Form validates input
6. Form submits to workflow
7. Workflow executes
8. Results shown to user

## Key Components

### Fields

Form fields collect different types of input:
- **Text**: Short text input
- **Textarea**: Long text
- **Email**: Email with validation
- **Number**: Integer or decimal
- **Checkbox**: Boolean true/false
- **Select**: Single choice dropdown
- **Multi-select**: Multiple choice
- **Date**: Date picker
- **File Upload**: Attach files

Each field has:
- Label (display text)
- Validation rules
- Help text
- Default value

### Data Providers

Populate dropdowns dynamically:

```python
@data_provider(name="get_departments")
async def get_departments(context):
    return [
        {"label": "Engineering", "value": "eng"},
        {"label": "Sales", "value": "sales"}
    ]
```

Used in form builder:
1. Add select field
2. Choose data provider
3. Dropdown populates automatically

### Visibility Rules

Show/hide fields based on other values:

```javascript
// Show only if checkbox checked
context.field.is_manager === true

// Show for specific role
context.field.role === "admin"

// Multiple conditions
context.field.role === "admin" && context.field.department === "engineering"
```

### Linked Workflow

Every form executes a workflow:
- Form collects parameters
- Submits to workflow
- Displays workflow result

```python
# Form submits to this workflow
@workflow(name="create_user", description="Create user")
@param("email", type="email", required=True)
@param("name", type="string", required=True)
async def create_user(context, email, name):
    return {"user_id": "123"}
```

## Form Flow

### 1. Form Load
- Fetch form definition
- Load data providers
- Populate initial values

### 2. User Interaction
- Fill fields
- Trigger visibility rules
- Real-time validation

### 3. Submission
- Validate all fields
- Prepare payload
- Call workflow API

### 4. Execution
- Workflow runs
- Logs captured
- Result returned

### 5. Display
- Show success/error
- Display result data
- Link to execution logs

## Form Types

### User Entry Forms
Collect info from end users:
- Service requests
- User onboarding

### Admin Forms
Management tasks:
- Bulk operations
- Configuration changes
- User management

### Automated Forms
Pre-filled for automation:
- Scheduled workflows
- Triggered workflows

## Field Mapping

Form fields → Workflow parameters:

**Form**:
```json
{
  "fields": [
    {"name": "email", "type": "email"},
    {"name": "department", "type": "select"}
  ]
}
```

**Workflow**:
```python
@workflow(name="create_user")
@param("email", type="email", required=True)
@param("department", type="string", required=True)
async def create_user(context, email, department):
    return {"user_id": "123"}
```

Field names must match parameter names.

## Dynamic Behavior

### Dependent Dropdowns

Cascade selections:

```python
# First dropdown
@data_provider(name="get_departments")
async def get_departments(context):
    return [{"label": "Sales", "value": "sales"}]

# Second dropdown depends on first
@data_provider(name="get_users_by_dept")
@param("department", type="string", required=True)
async def get_users_by_dept(context, department):
    users = await fetch_users(department)
    return [{"label": u.name, "value": u.id} for u in users]
```

User selects department → User dropdown updates.

### Conditional Required

Use visibility + validation:
1. Field not required by default
2. Visibility rule shows for specific condition
3. Workflow validates if condition met

### Real-Time Feedback

Forms validate as user types:
- Email format
- String length
- Number ranges
- Custom patterns

## Multi-Tenancy

Forms are org-scoped:
- Only org members see org forms
- Data providers filter by org
- Workflows execute with org context

Global forms (platform-wide) available to all orgs.

## Security

Forms respect permissions:
- User must have `canExecuteWorkflows` permission
- Workflow can require additional permissions
- Data providers filter by user permissions

## Best Practices

1. **Keep Simple**: Minimize fields, max clarity
2. **Use Data Providers**: Dynamic options over hardcoded
3. **Add Help Text**: Guide users with tooltips
4. **Validate Early**: Use field validation + workflow validation
5. **Test Flow**: Verify all visibility rules work
6. **Consider Mobile**: Forms work on all devices

## Forms vs Direct API

**Forms** (for users):
- Visual interface
- Guided input
- Validation feedback
- Result display

**API** (for automation):
- Direct workflow execution
- Programmatic access
- Bulk operations
- Integration

Both execute the same workflows - forms add UI layer.
