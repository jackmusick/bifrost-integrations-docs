Create AI agents that can use your workflows as tools and hold conversations.

## Overview

Agents combine:
- A system prompt defining personality/behavior
- Access to workflows as callable tools
- Optional knowledge sources for RAG
- Conversation history management

## Creating an Agent

<Steps>

1. Navigate to **Agents** in the sidebar

2. Click **Create Agent**

3. Configure:
   - **Name**: Display name for the agent
   - **System Prompt**: Instructions for the AI
   - **Access Level**: Public, Authenticated, or Role-based

4. Assign tools (workflows) the agent can call

5. Optionally add knowledge sources

</Steps>

## Agent Access Levels

| Level | Description |
|-------|-------------|
| **Public** | Anyone can chat (no auth required) |
| **Authenticated** | Logged-in users only |
| **Role-based** | Users with assigned roles only |

## Adding Tools

Mark workflows as tools to make them callable by agents:

```python
from bifrost import workflow

@workflow(
    is_tool=True,
    tool_description="Look up a customer by email address"
)
async def lookup_customer(email: str):
    # ... lookup logic
    return {"name": "John Doe", "account": "12345"}
```

The `tool_description` helps the agent understand when to use the tool.

## Knowledge Sources

Attach knowledge namespaces for RAG:

1. In agent settings, add **Knowledge Sources**
2. Select namespaces (e.g., "faq", "docs")
3. Agent automatically searches these when answering

## Chat API

Start conversations programmatically:

### Create Conversation

```python
import requests

response = requests.post(
    f"{api_url}/api/chat/conversations",
    json={"agent_id": "agent-123"},  # Optional
    headers={"Authorization": f"Bearer {token}"}
)
conversation_id = response.json()["id"]
```

### Send Message

```python
response = requests.post(
    f"{api_url}/api/chat/conversations/{conversation_id}/messages",
    json={"content": "Hello, I need help with my account."},
    headers={"Authorization": f"Bearer {token}"}
)
```

### WebSocket Streaming

For real-time streaming responses:

```javascript
const ws = new WebSocket(
    `wss://your-instance.com/api/chat/conversations/${conversationId}/stream`
);

ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data.type === "delta") {
        console.log(data.content);
    }
};

ws.send(JSON.stringify({ content: "How do I reset my password?" }));
```

## Agent Delegation

Agents can delegate tasks to other agents, enabling multi-agent workflows where specialized agents handle specific domains.

### Setting Up Delegation

<Steps>

1. In agent settings, add agents to **Delegated Agents**
2. The parent agent automatically gets a `delegate_to_<agent_name>` tool
3. During execution, the AI decides when to delegate based on context

</Steps>

### How Delegation Works

When an agent delegates:
1. A `delegate_to_<agent_name>` tool call is made with a `task` parameter
2. A **child agent run** is created and executed
3. The child run result is returned to the parent agent
4. Parent and child runs are linked via `parent_run_id` / `child_run_ids`

### Delegation Limits

| Limit | Value | Description |
|-------|-------|-------------|
| **Max depth** | 5 | Maximum delegation chain depth to prevent infinite loops |
| **Timeout** | 600s | Per-delegation timeout (10 minutes) |

<Aside type="tip">
Delegation is useful for specialized agents — a general triage agent can delegate to billing, tech support, or sales agents. Child runs are automatically filtered from the top-level agent runs list.
</Aside>

## Agentless Chat

Chat without a specific agent uses default routing:

```python
response = requests.post(
    f"{api_url}/api/chat/conversations",
    json={},  # No agent_id
    headers={"Authorization": f"Bearer {token}"}
)
```

The system uses AI-based routing to select an appropriate agent or responds with a default system prompt.

## Example: Support Agent

```python
from bifrost import workflow

@workflow(
    is_tool=True,
    tool_description="Look up a support ticket by ID"
)
async def get_ticket(ticket_id: str):
    # Fetch ticket from your system
    return {"id": ticket_id, "status": "open", "subject": "..."}

@workflow(
    is_tool=True,
    tool_description="Update the status of a support ticket"
)
async def update_ticket(ticket_id: str, status: str, note: str = ""):
    # Update ticket
    return {"success": True}
```

Create an agent with these tools and a prompt like:
> "You are a helpful support agent. Use the tools to look up and update tickets. Always confirm before making changes."

## Autonomous Agent Runs

Agents can run autonomously without a chat session — triggered by events, schedules, or SDK calls. The agent receives structured input, executes a tool-calling loop, and returns structured output.

### How It Works

1. An agent run is created with input data
2. The agent's system prompt and tools are loaded
3. The LLM processes the input and decides which tools to call
4. Tool results feed back into the LLM for the next iteration
5. The loop continues until the agent produces a final answer or hits budget limits

### Budget Controls

| Field | Description | Default | Max |
|-------|-------------|---------|-----|
| `max_iterations` | Maximum LLM call cycles | 50 | 200 |
| `max_token_budget` | Maximum total tokens consumed | 100,000 | 1,000,000 |
| `max_run_timeout` | Maximum seconds per run (optional) | None | — |

Set these on the agent configuration to control cost and runtime. If either `max_iterations` or `max_token_budget` is reached, the run completes with status `budget_exceeded`.

### Triggering Agent Runs

**From a workflow (SDK):**

```python
from bifrost import workflow, agents

@workflow
async def triage_ticket(ticket_id: str):
    result = await agents.run(
        "ticket-triage-agent",
        input={"ticket_id": ticket_id},
        output_schema={
            "type": "object",
            "properties": {
                "priority": {"type": "string"},
                "category": {"type": "string"}
            }
        },
    )
    return result
```

**From an event subscription:**

Create a subscription with `target_type: "agent"` to route events directly to an agent:

1. Go to **Events** > select your event source
2. Click **Add Subscription**
3. Set **Target Type** to **Agent**
4. Select the agent
5. The event payload becomes the agent's input

**From the API:**

```python
response = requests.post(
    f"{api_url}/api/agent-runs/execute",
    json={
        "agent_name": "ticket-triage-agent",
        "input": {"ticket_id": "T-123"},
        "output_schema": {"type": "object", "properties": {"priority": {"type": "string"}}},
        "timeout": 300
    },
    headers={"Authorization": f"Bearer {token}"}
)
result = response.json()  # {"output": {...}, "status": "completed", ...}
```

### Cancellation and Rerun

Agent runs can be cancelled or rerun from the UI or API:

- **Cancel**: Stops a running agent gracefully. The run transitions through `cancelling` → `cancelled`. Cancellation is checked at three layers: before each iteration, during tool execution, and via a background cancel watcher.
- **Rerun**: Re-executes a completed, failed, or cancelled run with the same input. Creates a new run with trigger `rerun`.

### Run Observability

Every autonomous run is tracked with full observability:

- **Steps**: Each LLM call and tool execution recorded (including delegation steps with expandable child run details)
- **Token usage**: Input/output tokens per step and totals
- **Cost tracking**: AI cost breakdown grouped by model in usage reports
- **Status progression**: `queued` → `running` → `completed` | `failed` | `budget_exceeded` | `cancelled`
- **Run details**: Iterations, tokens, duration, model, trigger source, and caller

View runs in the **History** page under the **Agents** tab, or query via the API.

## The Coding Agent

The Coding Agent is a platform agent for creating and modifying Bifrost workflows. It provides AI-assisted development capabilities directly within the platform.

### Characteristics

- **Claude-powered**: Uses Claude with MCP tools for code editing
- **Workspace access**: Has full access to workspace files and system tools
- **Development focus**: Optimized for workflow creation and modification
- **Precision editing**: Uses search → read → patch workflow for surgical code changes

### Precision Editing Tools

The Coding Agent uses precision editing tools that mirror professional code editor patterns:

| Tool | Description |
|------|-------------|
| `list_content` | List files by entity type (app_file, workflow, module) |
| `search_content` | Regex search across code with context lines |
| `read_content_lines` | Read specific line range from a file |
| `get_content` | Get entire file content |
| `patch_content` | Surgical string replacement (old → new) |
| `replace_content` | Full content replacement or create new file |
| `delete_content` | Delete a file (deactivates workflows, marks modules deleted) |

### Other System Tools

| Tool | Description |
|------|-------------|
| `execute_workflow` | Execute a workflow by name or ID |
| `list_workflows` | List all registered workflows |
| `validate_workflow` | Validate workflow Python syntax |
| `get_workflow` | Get workflow metadata |
| `get_workflow_schema` | Documentation about decorators |
| `list_executions` | List recent executions |
| `get_execution` | Get execution details and logs |
| `list_integrations` | List integrations and their status |
| `list_forms` | List all forms |
| `get_form` | Get form details |
| `get_form_schema` | Form structure documentation |
| `validate_form_schema` | Validate form JSON |
| `search_knowledge` | Search the knowledge base |
| `create_form` | Create a new form |
| `update_form` | Update an existing form |

## Making Workflows Available as Tools

Workflows can be exposed as tools for agents to call:

```python
from bifrost import workflow

@workflow(
    is_tool=True,
    tool_description="Fetch customer data by email"
)
async def get_customer(email: str):
    """Look up customer information."""
    # ... implementation
    return {"name": "John Doe", "account": "12345"}
```

When `is_tool=True`:
1. The workflow appears in the platform's tool list
2. Assign it to agents via the UI or API
3. The agent can then call the workflow as a tool during conversations

## Tool Assignment

Tools are assigned to agents in two ways:

- **System tools**: Platform-provided tools (file operations, workflow execution, search, etc.)
- **Workflow tools**: Assigned via the agent-tools relationship in the UI or API

Agents only receive the tools explicitly assigned to them.
