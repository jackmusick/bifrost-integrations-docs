## Why Contribute?

Bifrost exists to democratize automation tooling for the Integration Services industry. By contributing, you're helping to:

- **Build community-owned tools** that serve users, not investors
- **Prevent vendor lock-in** and extractive pricing models
- **Advance the Integration Services industry** with better tooling
- **Learn and grow** your skills in modern development practices

<Aside type="tip" title="This is our chance">
To own the tools we build our businesses on. Every contribution, no matter how small, helps ensure the next chapter of Integration Services doesn't get ravaged by venture capitalists.
</Aside>

## Getting Started (When Open)

Once contributions are open, here's how to get started:

<Steps>

1. **Fork the Repository**

   Fork the [bifrost-api](https://github.com/jackmusick/bifrost-api) or [bifrost-client](https://github.com/jackmusick/bifrost-client) repository to your GitHub account.

2. **Set Up Development Environment**

   Follow the [Local Development Guide](/sdk-reference/api/local-development) to get your environment ready.

3. **Create a Feature Branch**

   ```bash
   git checkout -b feature/amazing-feature
   ```

   Use descriptive branch names:
   - `feature/add-oauth-provider`
   - `fix/execution-timeout`
   - `docs/improve-quickstart`

4. **Make Your Changes**

   Write code, tests, and documentation.

5. **Run Quality Checks**

   ```bash
   # Run tests
   ./test.sh

   # Type check
   npx pyright

   # Lint
   ruff check .
   ```

6. **Commit Your Changes**

   Use clear, conventional commit messages:

   ```bash
   git commit -m "feat: add new OAuth provider support"
   git commit -m "fix: resolve execution timeout issue"
   git commit -m "docs: improve quick start guide"
   ```

7. **Push to Your Fork**

   ```bash
   git push origin feature/amazing-feature
   ```

8. **Open a Pull Request**

   Open a PR from your fork to the main repository.

   Include:
   - Clear description of changes
   - Link to related issues
   - Screenshots for UI changes
   - Migration notes if needed

</Steps>

## Code Standards

To maintain code quality and consistency:

### Python (API)

- **Style**: Follow PEP 8 style guide
- **Type Hints**: Use type hints for all functions
- **Docstrings**: Document public APIs with docstrings
- **Tests**: Write tests for new features
- **Linting**: Code must pass `ruff check .`
- **Type Checking**: Code must pass `npx pyright`

### TypeScript (Client)

- **Style**: Use Prettier for formatting
- **Types**: Enable strict mode, no `any` types
- **Components**: Follow React best practices
- **Tests**: Write tests for new components
- **Linting**: Code must pass `npm run lint`
- **Type Checking**: Code must pass `npm run tsc`

### Documentation

- **Clarity**: Write for beginners, not just experts
- **Examples**: Include code examples
- **Screenshots**: Add screenshots for UI features
- **Links**: Link to related documentation

## Development Workflow

### Before You Code

1. **Check existing issues** - Someone might already be working on it
2. **Open a discussion** - For large features, discuss first
3. **Get feedback** - Make sure your approach aligns with the project

### While Coding

1. **Commit frequently** - Small, focused commits
2. **Test locally** - Run tests before pushing
3. **Document changes** - Update docs as you go

### Before Submitting

1. **Run all tests** - `./test.sh` must pass
2. **Check types** - `npx pyright` must pass
3. **Lint code** - `ruff check .` must pass
4. **Update docs** - Document new features
5. **Write PR description** - Explain what and why

### PR Template

When opening a PR, include:

```markdown
## Description
Brief description of what this PR does

## Motivation
Why is this change needed?

## Changes
- List of specific changes
- Each change on its own line

## Testing
How was this tested?
- [ ] Unit tests pass
- [ ] Integration tests pass
- [ ] Manual testing completed

## Documentation
- [ ] Updated relevant documentation
- [ ] Added/updated code comments
- [ ] Updated CHANGELOG.md

## Screenshots (if applicable)
Add screenshots for UI changes

## Related Issues
Fixes #123
Related to #456
```

## Community Guidelines

### Be Respectful

- Treat everyone with respect and kindness
- Welcome newcomers and help them learn
- Assume good intentions
- Disagree constructively

### Be Collaborative

- Share knowledge and help others
- Give credit where credit is due
- Review others' PRs constructively
- Celebrate successes together

### Be Patient

- This is a community project, maintained by volunteers
- Reviews may take time
- Not all features will be accepted
- Be patient with yourself and others
